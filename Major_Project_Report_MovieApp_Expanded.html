ABSTRACT

The project "S-Show: Movie and TV Discovery Web Application" is a modern single-page web platform developed to provide a centralized, responsive, and user-friendly interface for entertainment data exploration. The system is implemented using React 18, Vite, Redux Toolkit, React Router, Tailwind CSS, Axios, and React Player, and consumes real-time data from The Movie Database (TMDB) API.

The core motivation behind this project is to reduce the complexity and fragmentation users face when searching for high-quality content information across different sources. Instead of navigating multiple websites, users can discover trending items, popular movies, TV shows, and people, filter by category and time duration, inspect metadata-rich detail pages, and watch official trailers in a single interface.

From a software engineering perspective, the project demonstrates a modular architecture with reusable components, route-based rendering, asynchronous API integration, and structured state management. The listing pages use infinite scroll for better browsing experience and reduced initial payload. Detail pages integrate multiple TMDB endpoints to consolidate recommendations, external IDs, translations, and video resources.

The report documents complete academic and technical details including problem context, literature analysis, objective mapping, methodology, architecture, data flow, implementation strategy, testing, result analysis, constraints, and future enhancements. The outcome is a scalable foundation for a production-grade content discovery platform.

Keywords: React, Vite, Redux Toolkit, TMDB API, Tailwind CSS, Infinite Scroll, Media Discovery

CHAPTER 1: INTRODUCTION

1.1 AIM

To design and develop a robust, interactive, and scalable web application that enables users to explore movies, TV shows, and celebrity profiles using real-time data, while ensuring a clean user interface, modular architecture, and maintainable codebase.

1.2 PROJECT BACKGROUND

Digital entertainment consumption has shifted from traditional broadcast systems to on-demand, recommendation-driven discovery ecosystems. Users now demand:

  • Fast search across multiple media categories
  • Concise and trustworthy metadata
  • High-visual layouts with quick previews
  • Direct trailer access
  • Multi-device compatibility

Large commercial platforms provide this functionality but often combine discovery with subscription systems, ads, account gates, and region restrictions. For students and developers studying modern web architecture, there is a need for a focused, lightweight, and technically transparent discovery platform.

The S-Show project addresses this need by providing:

  • Immediate access to trending and popular media
  • Category and duration filters
  • Rich detail pages driven by API aggregation
  • Modular front-end architecture suitable for learning and extension

CHAPTER 2: LITERATURE SURVEY

2.1 INTRODUCTION TO LITERATURE SURVEY

The literature survey evaluates existing systems, technical stacks, and design approaches used in modern media discovery platforms. The survey also identifies limitations in student-level implementations and maps opportunities for better architecture.

2.2 EXISTING SYSTEMS

2.2.1 OTT Platform Interfaces

Commercial OTT interfaces provide recommendation rails, watchlist management, and personalized feeds. However:

  • Data is often hidden behind account systems.
  • Discovery can be biased by subscription inventory.
  • Interfaces are optimized for retention, not necessarily transparent metadata exploration.

2.2.2 Movie Database Websites

Movie database portals provide rich data but may:

  • Present information in cluttered layouts.
  • Have inconsistent mobile-first design.
  • Offer less fluid navigation across categories.

2.2.3 Community Review Platforms

Community-driven systems are useful for reviews but less efficient for structured exploration and direct metadata comparisons.

2.3 STUDY OF TECHNOLOGIES USED

2.3.1 React

React supports component-driven development, declarative UI updates, and efficient re-rendering. It is suitable for highly interactive views such as dynamic cards, filters, and modal overlays.

2.3.2 Vite

Vite improves developer productivity with fast startup and hot module replacement. It is ideal for modern front-end projects requiring rapid iteration.

2.3.3 Redux Toolkit

Redux Toolkit simplifies store setup, reducers, and action handling. It helps maintain predictable state for detail pages where multiple API responses are aggregated.

2.3.4 React Router

React Router enables clean URL-based navigation and nested routes for detail/trailer flows.

2.3.5 Tailwind CSS

Tailwind allows utility-first styling, reducing context switching between CSS and JSX while promoting consistency.

2.3.6 Axios

Axios provides clean asynchronous HTTP request handling and supports centralized instance configuration with base URL and headers.

CHAPTER 4: PROPOSED METHODOLOGY

4.1 OVERALL METHODOLOGY

The project follows an iterative implementation methodology with continuous validation:

  1. Requirement Analysis
  2. System Design
  3. Module Implementation
  4. Integration Testing
  5. Documentation and Refinement

This approach allows incremental feature delivery and regular correction of design or implementation gaps.

4.2 SYSTEM ARCHITECTURE

4.2.1 High-Level Architecture

User -> React UI -> Route Layer -> Data Layer (Axios/Redux) -> TMDB API -> Rendered UI

4.2.2 Layer Description

  1. Presentation Layer: Components and pages.
  2. Navigation Layer: Router and nested routes.
  3. Data Access Layer: Axios client and async actions.
  4. State Layer: Redux slices for detail pages.
  5. External Service Layer: TMDB API endpoints.

4.3 COMPONENT-LEVEL DESIGN

The project is organized into:

  • Common partials (Topnav, Sidenav, Dropdown, Cards, Header)
  • Feature pages (Home, Trending, Popular, Movie, Tv_show, People)
  • Detail pages (Moviedetails, Tvdetails, Peopledetails)
  • Utility modules (Axios, store actions/reducers)

This modularization improves readability and future extension.

4.4 ROUTING DESIGN

Routing strategy includes:

  • Root home route
  • Separate listing routes per content type
  • Dynamic detail routes by content ID
  • Nested trailer routes for media details
  • Catch-all not-found route

4.5 DATA ACQUISITION STRATEGY

4.5.1 Listing Endpoints

  • Trending by media and duration
  • Category-based movie and TV endpoints
  • Popular people and popular media endpoints

4.5.2 Detail Aggregation Endpoints

For movies and TV, multiple endpoints are called:

  • Detail
  • External IDs
  • Recommendations
  • Videos
  • Similar
  • Translations
  • Watch providers

For people:

  • Detail
  • External IDs
  • Credits

4.6 STATE MANAGEMENT METHODOLOGY

A hybrid strategy is used:

  • Local component state for lists and pagination
  • Redux slices for complex detail pages

Benefits:

  • Keeps transient list state localized
  • Avoids overloading global store
  • Maintains consistent access to detailed media metadata

4.7 UI/UX METHODOLOGY

Design decisions include:

  • Dark-themed consistent visual language
  • Card-first content display
  • Prominent search bar and category filters
  • Back navigation support
  • Quick trailer action in detail views

4.8 PAGINATION AND SCROLL STRATEGY

Infinite scroll is implemented for better UX:

  • Initial page load fetches page 1
  • Additional pages requested on scroll threshold
  • hasMore flag controls pagination end condition

Advantages:

  • Reduced first paint payload
  • Better continuity of user browsing

4.9 ERROR AND LOADING HANDLING

Current strategy:

  • Loading component for delayed content
  • Console logging of fetch errors
  • Fallback views where needed

Future strategy:

  • Dedicated error boundaries
  • Retry actions
  • User-visible status messages

4.10 PHYSICAL DESIGN

  • Browser-based client application
  • Built with Vite static bundling
  • No server process required in final static deployment mode

4.11 DATABASE DESIGN

No internal database in current version.
Data model is virtualized from TMDB responses and represented through JavaScript objects.

Appendix H: Detailed Screen-Wise Technical Description

H.1 Home Screen Behavior

The home screen acts as the landing dashboard and combines a curated hero section with trending content preview cards. On first render, two independent flows are executed:

  1. Header spotlight fetch (/trending/all/day) to select randomized banner media.
  2. Trending feed fetch (/trending/{category}/day) to populate horizontal card rail.

The screen demonstrates the following engineering principles:

  • Multi-fetch initialization
  • Conditional rendering (Loading while data is absent)
  • Filter-driven re-fetch behavior through Dropdown
  • Reusable partial composition (Sidenav, Topnav, Header, HorizortalCards)

H.2 Trending Screen Behavior

The trending screen provides category (movie, tv, all) and duration (day, week) controls. This screen also integrates infinite scrolling and therefore represents one of the most interaction-heavy pages in the project.

Key runtime cycle:

  1. Category or duration update triggers list reset.
  2. Page counter resets to 1.
  3. First page data is fetched.
  4. Scroll threshold triggers next callback.
  5. New page appends to existing collection.
  6. hasMore toggles false when exhausted.

This behavior ensures stable pagination without full reloads.

H.3 Popular Screen Behavior

The popular module showcases dynamic endpoint switching based on selected category. Instead of maintaining separate pages for popular movies and TV shows, the module interpolates endpoint paths using selected category state.

Benefits:

  • Less route duplication
  • Reduced UI and API logic redundancy
  • Better maintainability

H.4 Movie Screen Behavior

The movie screen exposes popular categories (popular, top_rated, upcoming, now_playing) and supports continuous browsing. The module uses local state because list state is temporary and does not require global store persistence.

H.5 TV Screen Behavior

The TV module mirrors movie behavior with TV-specific categories (airing_today, on_the_air, popular, top_rated). This symmetry is useful for code standardization and future abstraction of list behavior into custom hooks.

H.6 People Screen Behavior

The people screen fetches popular persons and supports infinite scrolling. This module complements media discovery by letting users navigate from person profiles to related filmography context.

H.7 Movie Detail Screen Behavior

The movie detail screen performs data aggregation from multiple endpoints and dispatches one combined payload into Redux state. This allows a consolidated render model including:

  • Core movie details
  • External references
  • Trailer metadata
  • Recommendation and similar items
  • Translation values
  • Watch provider data

The cleanup process (removemovie) runs on unmount to avoid stale cross-item state.

H.8 TV Detail Screen Behavior

The TV detail screen closely follows movie detail architecture and includes nested trailer route behavior. It demonstrates reuse of structure and style while adapting endpoint paths and date fields (first_air_date vs release_date).

H.9 People Detail Screen Behavior

The person detail screen provides:

  • Personal profile metadata
  • Social platform links via external IDs
  • Biography and identity descriptors

The action combines person details and credits into a single store payload for clean component consumption.

H.10 Trailer Overlay Behavior

Trailer rendering is route-driven using nested detail route patterns. The component inspects current path to identify content type (movie or tv) and pulls trailer video from corresponding Redux slice.

User experience benefits:

  • No full navigation away from detail context
  • Fast close/back interaction
  • Consistent watch flow

Appendix I: Algorithmic Logic and Pseudocode

I.1 Infinite Scroll Data Loader

Initialize list = []
Initialize page = 1
Initialize hasMore = true

Function resetList():
    list = []
    page = 1
    hasMore = true
    fetchList(page, isReset=true)

Function fetchList(pageToFetch, isReset):
    response = call endpoint with page = pageToFetch
    if response.results has elements:
        if isReset:
            list = response.results
        else:
            list = list + response.results
        page = pageToFetch + 1
    else:
        hasMore = false

I.2 Detail Aggregation Logic

Function loadMovieDetails(id):
    detail = GET /movie/{id}
    external = GET /movie/{id}/external_ids
    rec = GET /movie/{id}/recommendations
    videos = GET /movie/{id}/videos
    similar = GET /movie/{id}/similar
    trans = GET /movie/{id}/translations
    providers = GET /movie/{id}/watch/providers

    payload = {
        detail: detail.data,
        externalid: external.data,
        recommendations: rec.data.results,
        videos: first Trailer from videos.results,
        similar: similar.data.results,
        watchproviders: providers.data.results,
        translations: map(translations -> name)
    }

    dispatch(loadmovie(payload))

Appendix N: Viva Preparation (Extended)

N.1 Architecture Questions

Q1. Why choose a client-side only architecture for this project phase?
Because project scope prioritizes front-end discovery workflows, API integration, and academic demonstration of modern SPA architecture without backend persistence overhead.

Q2. Why not keep all data in Redux?
List pages contain transient pagination state that is locally relevant. Globalizing it would increase store complexity and boilerplate without clear benefit.

Q3. Why use nested routes for trailer?
Nested routes preserve context of detail pages, enabling overlay-like behavior and intuitive back navigation.

N.2 Data and API Questions

Q4. How do you avoid repeated request configuration?
By using a centralized Axios instance with base URL and authorization headers.

Q5. How is multi-endpoint detail data handled?
Each action gathers related endpoint responses and dispatches a unified payload, simplifying UI rendering.

N.3 Quality and Security Questions

Q6. Biggest security concern in current implementation?
Hardcoded API bearer token in client source.

Q7. Immediate fix?
Use environment variables and keep sensitive values out of repository.

Q8. How to improve reliability?
Add explicit error boundaries, retries, and automated test layers.

N.4 Future Roadmap Questions

Q9. What would be first production enhancement?
Authentication + persistent watchlist with backend service.

Q10. What optimization would you prioritize next?
Route-level code splitting and API response caching.
